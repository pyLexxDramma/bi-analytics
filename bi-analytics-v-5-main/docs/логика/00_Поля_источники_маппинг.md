# Поля интерфейса, источники FTP и маппинг

Документ сделан как техническое описание кода: что берется с FTP, как парсится и во что маппится в приложении.

Формат каждой строки:
1. Название поля в интерфейсе
2. Источник файла из FTP
3. Колонка в FTP + canonical/admin key (куда маппится)

---

## 1) MSP (сроки/причины/девелоперка/контрольные точки)

Источник FTP: `web/AI/msp_*_<date>.csv`

| Поле в интерфейсе | Колонка FTP | Canonical/Admin key |
|---|---|---|
| Проект | `ID_проекта` / `Проект` | `project id` / `project name` |
| Лот | `ЛОТ` | `lot` |
| Блок | `БЛОК` | `block` |
| Раздел (иерархия) | родитель по структуре задачи | `section` (восстанавливается из дерева задач) |
| Задача | `Название задачи` / `Название` | `task name` |
| Уровень | `Уровень` | `level` |
| Уровень структуры | `Уровень_структуры` | `level structure` |
| План: начало | `Начало` / `План начало` | `plan start` |
| План: окончание | `Окончание` / `План окончание` | `plan end` |
| База: начало | `Базовое_начало` / `Базовое начало` | `base start` |
| База: окончание | `Базовое_окончание` / `Базовое окончание` | `base end` |
| Факт окончания | `Фактическое_окончание` | `actual finish` |
| Причина отклонения | `Причины_отклонений` | `reason of deviation` |
| Отклонение окончания (дни) | `Отклонение_окончания` | `deviation in days` |
| Отклонение начала (дни) | `Отклонение_начала` | `deviation start days` |
| % завершения | `Процент_завершения` | `pct complete` |
| Шифр ПД/РД | `Шифр_ПД_и_РД` | `abbreviation` |
| Предшественники | `Предшественники` | `predecessors` |
| Последователи | `Последователи` | `successors` |
| Примечание | `Заметки` | `notes` |

### Логика парсинга
- Применяется `_MSP_COLUMN_REMAP` в `web_loader.py`.
- Даты приводятся к `datetime`, числовые отклонения — к числам.
- Для проекта оставляется последний `snapshot_date` (последний снимок).
- `section` заполняется по иерархии MSP (родитель уровня 2), а не из `ЛОТ`.

### Что выводится
- Сроковые дашборды: причины, отклонения от базового плана, график проекта, контрольные точки.
- Девелоперские сводки по проектам/блокам/этапам.

## 2) 1С `dannye.json` (БДДС/БДР/бюджеты)

Источник FTP: `web/*_dannye.json`

| Поле в интерфейсе | Колонка FTP | Canonical/Admin key |
|---|---|---|
| Период | `Период` | `period` |
| Сценарий | `Сценарий` | `scenario` |
| Статья оборотов | `СтатьяОборотов` | `turnover item` |
| Проект | `Проект` | `project name` |
| Сумма | `Сумма` | `amount` |
| Доход/Расход | `РасходДоход` | `income_expense_flag` |
| Тип статьи | `ТипСтатьи` | `item_type` |

### Логика парсинга
- JSON нормализуется в табличный вид.
- Числовые поля очищаются от строкового формата и приводятся к `float`.
- Для режима “накопительно” считается кумулятивная сумма.

### Что выводится
- БДДС по месяцам/накопительно.
- БДР по месяцам/накопительно.
- Бюджетные срезы plan/fact/forecast.

## 3) 1С `DK.json` (дебиторка/кредиторка)

Источник FTP: `web/*_DK.json`

| Поле в интерфейсе | Колонка FTP | Canonical/Admin key |
|---|---|---|
| Организация | `Организация` | `organization` |
| Контрагент | `Контрагент` | `contractor` |
| Договор | `Договор` | `contract` |
| Остаток на начало | `ОстатокНаНачалоПериода` | `opening_balance` |
| Начало по авансам | `ОстатокНаНачалоПериодаПоАвансам` | `opening_advance_balance` |
| Всего оплат | `ВсегоОплат` | `total_payments` |
| Оплаты авансом | `ВсегоОплат_Аванс` | `total_advance_payments` |
| Остаток на конец | `ОстатокНаКонецПериода` | `closing_balance` |
| Конец по авансам | `ОстатокНаКонецПериодаПоАвансам` | `closing_advance_balance` |

### Логика парсинга
- Парсинг JSON в таблицу по контрагентам/договорам.
- Для нетто-остатков используется вычитание авансовых частей из общих остатков.

### Что выводится
- KPI и таблицы дебиторки/кредиторки подрядчиков.
- Динамика остатков и оплат.

## 4) 1С `spravochniki.json` (справочники и план ГДРС)

Источник FTP: `web/*_spravochniki.json`

| Поле в интерфейсе | Колонка FTP | Canonical/Admin key |
|---|---|---|
| Контрагент (план ГДРС) | `Наименование_Контрагента` | `contractor_name` |
| Проект (план ГДРС) | `Наименование_Проекта` | `project_name` |
| План: рабочие | `КоличествоРаботников` | `gdrs_plan_workers` |
| План: техника | `КоличествоСпецТехники` | `gdrs_plan_equipment` |

### Логика парсинга
- Поля плана могут быть массивом периодов `[{Дата, Количество}]`.
- Для сравнения с фактом агрегируются по текущему периоду фильтра.
- Связка с фактом: по нормализованному контрагенту и проекту.

### Что выводится
- В `ГДРС`: план/факт рабочие.
- В `ГДРС Техника`: план/факт техника.

## 5) TESSA ID (исполнительная документация и предписания)

Источник FTP: `web/tessa_*_id.csv`

| Поле в интерфейсе | Колонка FTP | Canonical/Admin key |
|---|---|---|
| Карточка документа | `DocID` | `doc_id` |
| Тип документа | `KindName` | `kind_name` |
| Статус KR | `KrState` / `KrStateID` | `kr_state` / `kr_state_id` |
| Дата создания | `CreationDate` | `created_at` |
| Проект/объект | `ObjectName` | `object_name` |
| Лот | `Lot` | `lot` |
| Номер документа | `DocNumber` | `doc_number` |
| Описание | `DocDescription` | `doc_description` |

### Логика парсинга
- Загрузка CSV с нормализацией кодировки/разделителя.
- Статусы расшифровываются через `KrStates.csv`.

### Что выводится
- Исполнительная документация: KPI по статусам, таблицы карточек.
- Предписания: статус устранения, просрочки, детализация.

## 6) TESSA TASK (связанные задачи/результаты)

Источник FTP: `web/tessa_*_task.csv`

| Поле в интерфейсе | Колонка FTP | Canonical/Admin key |
|---|---|---|
| Связь с карточкой | `CardID` | `card_id` |

| Опция/решение | `OptionCaption` | `task_option` |

### Что выводится
- Дополнительные признаки статуса в ИД и предписаниях.

## 7) TESSA RD + План РД (рабочая/проектная документация)

Источники FTP:
- Факт: `web/tessa_*_rd.csv`
- План: `web/AI/other_*_rd.csv`

| Поле в интерфейсе | Колонка FTP | Canonical/Admin key |
|---|---|---|
| Проект | `ObjectName` / `ObjectProjectName` | `project name` |
| Шифр раздела | `DivisionCipher` / `Шифр` | `division_cipher` |
| Раздел | `SubDivisionVersionName` / `Наименование раздела` | `section_name` |
| Плановая дата выдачи | `Дата выдачи разделов по Договору` | `rd_plan_date` |
| Кол-во разделов по договору | `Количество разделов по Договору` | `rd_plan_count` |
| Фактическая дата/статус | `CreationDate`, `KrState` | `rd_fact_date`, `rd_status` |

### Логика парсинга
- План/факт связываются по ключу `проект + шифр + раздел`.
- Просрочка = разница между фактической и плановой датой.

### Что выводится
- Отчеты “Рабочая документация” и “Проектная документация”.
- План/факт по срокам и таблица отклонений.

## 8) Факт ресурсов (ГДРС/СКУД)

Источник FTP: `web/AI/other_*_resursi.csv`

| Поле в интерфейсе | Колонка FTP | Canonical/Admin key |
|---|---|---|
| Проект | `Проект` | `project name` |
| Подрядчик | `Подрядчик` / `Контрагент` | `contractor` |
| Тип ресурса | `тип ресурсов` | `resource_type` |
| Источник строки | `data_source` (если есть) | `data_source` |
| Значения по дням | колонки дат `ДД.ММ.ГГГГ` | `daily_values` |
| Значения по неделям | `N неделя` (если есть) | `weekly_values` |

### Логика парсинга
- Строки разделяются на “рабочие” и “техника”.
- Дневные колонки агрегируются в недельные/месячные срезы.
- Далее сравниваются с планом из `spravochniki.json`.

### Что выводится
- `ГДРС` (рабочие): план/факт, доли подрядчиков, таблица.
- `ГДРС Техника`: план/факт техники и отклонения.

## Техническая привязка к коду
- Основной маппинг MSP: `web_loader.py` → `_MSP_COLUMN_REMAP`.
- Расширенный маппинг русских/legacy названий: `data_loader.py` → `column_mapping`.
- Реестр отчетов и названия в интерфейсе: `dashboards/__init__.py`.
- Детальная логика отрисовки по дашбордам: `dashboards/_renderers.py`.
