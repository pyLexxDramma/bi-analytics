# Процесс работы по ТЗ заказчика и предупреждениям на дашбордах

Согласованный цикл: скрин ТЗ → сверка с кодом и правки → повторная сверка со скрином и файлами из `web/` → формулировки предупреждений → реализация → визуальная приёмка → после «ок» коммит и пуш в `main`.

## Контекст в коде

| Назначение | Путь |
|------------|------|
| Fallback и синтетика из 1С | `dashboards/finance_from_1c.py` |
| Отрисовка дашбордов | `dashboards/_renderers.py` |
| Централизованные тексты предупреждений | `dashboards/data_quality_hints.py` |
| Загрузка выгрузок, восстановление `reference_1c_dannye` | `web_loader.py` (корень пакета приложения) |
| Эталонные JSON/CSV для сверки | каталог `web/` |

## Цикл (шаги 1–7)

1. **Скрин фрагмента ТЗ** — раздел, таблица, подписи осей, формулы отклонений, единицы и т.д.
2. **Сверка с реализацией** — соответствующий дашборд/функция в `_renderers.py`, при необходимости цепочка в `finance_from_1c`, `web_loader`, `utils`. Правки под ТЗ при расхождениях.
3. **Повторный скрин ТЗ + `web/`** — колонки, сценарии, агрегации против примеров (`*dannye.json`, MSP CSV и др.). Фиксация расхождений «ТЗ ↔ код ↔ данные». Указывать **имя файла** из `web/` или фрагмент.
4. **Тексты предупреждений** — при дырах или неверном формате: когда показывать, по каким флагам (`attrs`, fallback, пустые агрегаты). Шаблон см. ниже.
5. **Реализация** — расширить `data_quality_hints.py`, точечно вызывать `render_quality_hints` в `_renderers.py` или выставить флаги при подготовке данных.
6. **Визуальная приёмка** заказчиком в приложении.
7. **После «ок»** — один осмысленный коммит, `git push origin main` из корня репозитория; в индекс только целевые файлы (без массовых артефактов из `web/` и временных скриптов, если не нужны).
