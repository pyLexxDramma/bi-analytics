# Деплой: переменные окружения для dev и release

**Версия:** 2026-05-05 (обновлено: admin-режим)
**Цель:** Гарантировать, что после деплоя на ветку `dev` и на ветку `release` приложение подтягивает **только реальные свежие выгрузки** (MSP, 1С, TESSA из `web/`), без демо-данных и без dev-диагностики на стороне клиента.

---

## TL;DR

| Сценарий | Что задать на сервере |
|----------|------------------------|
| **Release / клиент (production)** | `BI_ANALYTICS_HIDE_DEV_DIAGNOSTICS=1` (или `BI_ANALYTICS_RELEASE_MODE=1`) |
| **Dev / разработка** | ничего не задавать — по умолчанию демо **выключены** (как клиент видит) |
| **Dev / включить демо для UI-демонстрации** | admin кликает тумблер «Подмешивать демо-данные» → вкл (или `BI_ANALYTICS_INCLUDE_DEMO=1` для автоматического включения на старте) |

Опционально (любой режим):
- Полная история всех снимков в `web/` (вместо последнего): `BI_ANALYTICS_WEB_LATEST_ONLY=0`.
- Дополнительные каталоги для сканирования: `BI_ANALYTICS_WEB_EXTRA_PATHS="C:\path1;C:\path2"` (`;` или `,` как разделитель).
- **Авто-ingest при холодном старте инстанса** (Streamlit Cloud / прод): `BI_ANALYTICS_AUTO_INGEST=1`. Подробно — см. раздел [«Авто-ingest при старте»](#авто-ingest-при-старте) ниже. Решает проблему «после перезапуска инстанса web_data.db эфемерного диска пустая → дашборды без данных, пока админ не нажмёт «Загрузить из web/»».

Перезапуск процесса Streamlit после смены любой переменной обязателен.

---

## Что меняется в release-режиме

Все правила собраны в `config.py · is_release_client_mode()`. Этот предикат `True`, если выполнено **любое** из:
- задана `BI_ANALYTICS_HIDE_DEV_DIAGNOSTICS=1` (`true`/`yes`/`on`);
- задана `BI_ANALYTICS_RELEASE_MODE=1`;
- текущая git-ветка приложения = `release` (определяется автоматически).

Когда `True`:

1. **Не подмешиваются демо-файлы** в загрузку из `web/`:
   - все `sample_*.csv` в `web/` игнорируются;
   - папка `new_csv/` рядом с приложением игнорируется;
   - любой файл внутри подкаталога `new_csv/` (где бы то ни было в дереве `web/`) игнорируется.
   Реализация: `web_loader._is_demo_file()` + `scan_web_files()` + `load_all_from_web()` (эти места читают `config.ignore_demo_data_files()`, который теперь автоматически `True` в release-режиме).

2. **Скрыта dev-диагностика** в UI:
   - зелёный `«Загружено файлов: N»`,
   - жёлтый `«Режим последних снимков: файлов было …»`,
   - expander `«Справка: колонки загрузки из web/»`.
   Сообщения об **ошибках** и о **пропущенных** файлах остаются — это критично для эксплуатации.

3. **Режим «Загрузить вручную»** скрыт от обычных пользователей. **Доступен только ролям `admin` / `superadmin`** — как страховка на случай отказа FTP/web/. Реализация: `project_visualization_app.py` строит `data_mode_options` в зависимости от `has_admin_access(user.role)`.

4. **Авто-перечитывание `web/` при первом заходе клиента**, если БД пуста (одноразово в сессии: `_release_web_autoload_tried`). Если на сервере настроен FTP (`BI_FTP_HOST`+`BI_FTP_USER` либо `st.secrets[ftp]`), перед перечтением `web/` **автоматически выполняется FTP-синхронизация** — т.е. клиент при первом заходе сразу получает свежие данные с FTP без участия админа. Отключить это поведение: `BI_ANALYTICS_AUTO_FTP_ON_START=0`.

5. **Admin-тумблер «Подмешивать демо-данные» в сайдбаре** — на release **скрыт** (даже у админа). Демо на release включить нельзя ни через сессию, ни через `BI_ANALYTICS_INCLUDE_DEMO=1`. Это аппаратное правило.

## Защитный фильтр на стороне дашборда «Девелоперские проекты»

Даже если в активной версии БД остались старые sample-данные (например, БД была наполнена ещё до перехода на release-режим), дашборд **«Девелоперские проекты»** дополнительно отсекает строки с `__source_file LIKE 'sample_%'` (и `new_csv/sample_%`) — см. `dashboards/_renderers.py · dashboard_developer_projects`. Это безусловный страховочный слой.

## Полный список переменных окружения

| Переменная | Значения | Эффект |
|------------|----------|--------|
| `BI_ANALYTICS_HIDE_DEV_DIAGNOSTICS` | `1`/`true`/`yes`/`on` | Включает release-режим (см. выше). Рекомендуется для production-деплоя. |
| `BI_ANALYTICS_RELEASE_MODE` | `1`/`true`/`yes`/`on` | Синоним `BI_ANALYTICS_HIDE_DEV_DIAGNOSTICS`. |
| `BI_ANALYTICS_IGNORE_DEMO` | `1`/`true`/`yes`/`on` | На dev: принудительно игнорировать демо. На release не нужна — там игнор автоматический. |
| `BI_ANALYTICS_INCLUDE_DEMO` | `1`/`true`/`yes`/`on` | На dev: явно включить демо (не нужна, т.к. dev и так включает). **Не работает на release** — там аппаратный запрет. |
| `BI_ANALYTICS_WEB_LATEST_ONLY` | `0`/`1` (default `1`) | `0` — грузить **все** даты в `web/`, `1` — только последний снимок по дате в имени. |
| `BI_ANALYTICS_WEB_EXTRA_PATHS` | `path1;path2` или `path1,path2` | Дополнительные корни для сканирования CSV/JSON. |
| `BI_FTP_HOST`, `BI_FTP_USER`, `BI_FTP_PASSWORD` (или `FTP_AI_PASSWORD`), `BI_FTP_REMOTE_DIR` | строки | Доступ к FTP для кнопки «FTP → web/» **и** для авто-FTP при первом заходе клиента на release. На release-сервере удобнее задать через secrets / env. Дополнительно: `BI_FTP_PORT` (по умолчанию `21`), `BI_FTP_TLS=1` (для FTPS). |
| `BI_ANALYTICS_AUTO_FTP_ON_START` | `0`/`false`/`no`/`off` | Отключить авто-FTP при первом заходе клиента на release. По умолчанию (если FTP настроен) авто-FTP **включена**. |
| `BI_FTP_RECURSIVE` | `0`/`false`/`no`/`off` (default `1`) | По умолчанию `sync_ftp_to_web` обходит **подпапки** на FTP с сохранением структуры (нужно для `/web/AI/msp_*.csv`). `0` — только корень `BI_FTP_REMOTE_DIR` (старое поведение). |
| `BI_FTP_FORCE_REDOWNLOAD` | `1`/`true` (default `0`) | По умолчанию включён инкрементальный режим: перед `RETR` сравниваем удалённый размер (MLSD/SIZE) с локальным; совпало — пропускаем. `1` — игнорировать проверку и качать заново все файлы. |

### Приоритет источников решения для `ignore_demo_data_files()`

Проверяется сверху вниз, первый сработавший побеждает:

1. **release-режим** → `True` (демо отключены, нельзя обойти).
2. **Сессионный admin-тумблер** `_admin_demo_pref` (только в dev): `include` → `False`, `ignore` → `True`.
3. **ENV** `BI_ANALYTICS_INCLUDE_DEMO=1` → `False`.
4. **ENV** `BI_ANALYTICS_IGNORE_DEMO=1` → `True`.
5. **Дефолт**: `True` — демо игнорируются. Безопасный по умолчанию подход: разработчик видит данные так же, как клиент; чтобы включить демо, нужно явное действие admin.

## Чек-лист деплоя на release

1. На сервере (Streamlit Cloud / Docker / systemd) задать:
   ```
   BI_ANALYTICS_HIDE_DEV_DIAGNOSTICS=1
   BI_ANALYTICS_WEB_LATEST_ONLY=1     # (или не задавать — это значение по умолчанию)
   ```
   **НЕ задавать** `BI_ANALYTICS_INCLUDE_DEMO` (даже если задать — на release не сработает).

2. Если есть FTP с актуальными выгрузками — задать `BI_FTP_*` и в UI один раз нажать «FTP → web/ → Скачать ... и загрузить в БД». Дальше клиент может обновлять данные сам.

3. Как клиент обновляет данные сам (без участия разработчика):
   - **Основной путь** — кнопка «FTP → web/» в шапке отчётов (если задан FTP).
   - **Альтернатива** — «Из папки web/ → Загрузить из web/», если файлы в `web/` доставлены другими средствами (scp/RDP/scheduled task).
   - **Ручная загрузка** — у клиента **нет**. Если нужно срочно подсунуть файл, заходит admin: ему доступен дополнительный пункт радио «Загрузить вручную».

4. Убедиться, что в каталоге `web/` (и в любом из `BI_ANALYTICS_WEB_EXTRA_PATHS`) **нет** `sample_*.csv` файлов и нет подкаталогов `new_csv/`. Если такие файлы случайно попали — release-режим их всё равно проигнорирует, но лучше очистить.

5. Открыть приложение под клиентским пользователем (роль `manager`/`analyst`/etc., не admin) → меню «Девелоперские проекты» → проверить:
   - в фильтре «Проект» **только реальные** проекты (`Дмитровский 1`, `Есипово V`, `Ленинский`), без `Завод` / `Дмитровский` (без `1`);
   - в радио «Источник данных» только `Из папки web/` и `FTP → web/`;
   - в сайдбаре **нет** блока «Dev-инструменты».

6. Открыть приложение под admin → проверить:
   - в радио источника появилось «Загрузить вручную» (3 пункта);
   - в сайдбаре **нет** блока «Dev-инструменты» (он скрыт даже у admin на release).

7. Если в матрице проекты всё ещё включают демо — значит активна **старая** версия БД, наполненная до деплоя release-флага. На сервере: нажать «Загрузить из web/» (создаст новую чистую активную версию) или сделать FTP-синхронизацию. Старые версии в БД не удаляются автоматически — очистить можно через админ-панель «Параметры отчётов» либо вручную SQL по таблицам `web_versions` / `web_files` / `web_data` в `data/web_data.db`.

## Единый источник данных для local / dev cloud / release cloud (через FTP)

Цель: чтобы любой пользователь — разработчик локально, тестировщик на dev и клиент на release — видел **один и тот же набор файлов** и одинаковые цифры в дашбордах. Решение: один FTP-источник для всех трёх окружений.

### 1. На FTP-сервере (источник истины)

Залить актуальные:
- `msp_*_DD-MM-YYYY.csv` (MSP-выгрузки по проектам);
- `1с_DD-MM-YYYY_HH-MM_*.json` (выгрузки 1С);
- `tessa_DD-MM-YYYY-HH-MM-{rd,task,id,razdel}.csv` (TESSA);
- любые `other_*` / справочники.

Имена должны содержать дату — иначе `pick_latest_snapshot_files` будет выбирать «свежий» по `mtime` (хуже, но работает).

### 2. На release-сервере (Streamlit Cloud → Settings → Secrets)

```toml
[ftp]
host = "ftp.example.ru"
user = "bi-client"
password = "***"
remote_dir = "/web"
# port = 21              # опционально
# use_tls = true         # опционально, если FTPS
```

И env (Settings → Secrets → переменные окружения, если интерфейс Cloud это поддерживает, — иначе через TOML с верхнего уровня):

```toml
BI_ANALYTICS_RELEASE_MODE = "1"
# BI_ANALYTICS_AUTO_FTP_ON_START = "0"  # раскомментировать, если авто-FTP при старте не нужен
```

Эффект:
- При первом заходе клиента в фоне выполняется `ftp_sync.sync_ftp_to_web(web/, …)`.
- Затем `load_all_from_web()` создаёт активную версию БД из свежих файлов.
- Клиент видит данные сразу, без ручных кнопок и без диагностики.

#### Инкрементальная скачка и подпапки

Скачивание устроено так, чтобы повторный «Скачать с FTP» отрабатывал быстро:

- **Рекурсия по подпапкам** (`BI_FTP_RECURSIVE=1`, default). На FTP MSP-файлы лежат в `/web/AI/`. Старая плоская реализация их не подтягивала — поэтому на cloud была активная версия только на `21` файл, а локально была на `26`. Теперь обходим все подпапки с сохранением структуры (`web/AI/msp_*.csv`).
- **Инкрементальная проверка размера** (`BI_FTP_FORCE_REDOWNLOAD=0`, default). Перед `RETR` берём размер удалённого файла из MLSD (или `SIZE`); если у локального копия такой же размер — пропускаем. На стенде «409 файлов уже на диске» повторный `python -m ftp_sync` выполняется за **~4 секунды** (вместо нескольких минут полной перекачки).
- **Атомарная запись через `*.tmp`**: RETR пишет во временный файл, и только после успеха `os.replace` переименовывает поверх. Если на FTP-сервере прямо сейчас пишет 1С и `RETR` упал с `550 Failed to open file`, локальный валидный файл **не повреждается**.

Принудительная полная перекачка (если кажется, что что-то «подвисло»): задайте `BI_FTP_FORCE_REDOWNLOAD=1` и нажмите «Скачать…» один раз, затем верните в `0`.

### 3. На dev-сервере (Streamlit Cloud для тестировщика)

То же `[ftp]`, но **без** `BI_ANALYTICS_RELEASE_MODE`. Отличия от release:
- видны зелёные/жёлтые сообщения о загрузке;
- admin/superadmin видит блок «Dev-инструменты» с тумблером «Подмешивать демо-данные» (по умолчанию OFF);
- доступен пункт «Загрузить вручную» для всех ролей.

Авто-FTP при первом заходе клиента — **release-only**. Чтобы dev сам подтягивал данные при холодном старте инстанса, добавь общий флаг `BI_ANALYTICS_AUTO_INGEST=1` (см. ниже). Иначе админ нажимает «FTP → web/» → «Скачать CSV и JSON с FTP в web/ и загрузить в БД».

## Авто-ingest при старте

**Проблема.** Streamlit Cloud-инстансы имеют **эфемерный диск** — `web_data.db` теряется при холодном перезапуске инстанса. Без авто-ingest приложение поднимается с пустой БД, и пока админ не нажмёт «Загрузить из web/» (или клиент на release не запустит first-touch авто-FTP), дашборды пустые.

**Решение.** Включить флаг `BI_ANALYTICS_AUTO_INGEST=1` (на dev и/или release). При первом холодном старте процесса:

1. (опционально) `ftp_sync.sync_ftp_to_web(web/, …)` — подтянет свежие файлы.
2. `web_loader.load_all_from_web()` — создаст SUCCESS-версию в БД, она автоматически становится «активной» через `get_active_version_id()`.

Маркер успеха пишется на эфемерный диск рядом с `web_data.db` → при пересоздании инстанса ingest повторится автоматически. Реализация: `auto_ingest.py · maybe_run_auto_ingest_on_startup()`, вызывается в `project_visualization_app.py` сразу после `init_db()`.

### Переменные окружения

| Переменная | Default | Эффект |
|------------|---------|--------|
| `BI_ANALYTICS_AUTO_INGEST` | `0` | Мастер-флаг. `1` → включить авто-ingest при старте. |
| `BI_ANALYTICS_AUTO_INGEST_FTP` | `1` | `1` → перед `load_all_from_web` сделать FTP-sync (требует BI_FTP_HOST/USER). `0` → только из `web/`. |
| `BI_ANALYTICS_AUTO_INGEST_AGE_H` | `12` | Не повторять чаще раза в N часов (защита от лишних рестартов в Streamlit Cloud). `0` → каждый раз при холодном старте. |
| `BI_ANALYTICS_AUTO_INGEST_FORCE` | `0` | `1` → игнорировать маркер, ingest каждый старт (для отладки). |
| `BI_ANALYTICS_AUTO_INGEST_PURGE_DB` | `1` | `1` → при смене `app_version` (см. ниже) удалять `web_data.db` перед ingest, чтобы заново построить SUCCESS-версию из текущих `web/*` файлов. |
| `BI_ANALYTICS_BUILD_VERSION` | (auto) | Явно фиксированный sha сборки. Если не задано — берётся из `git rev-parse --short HEAD`, иначе из `_build_info.json`, иначе mtime `project_visualization_app.py`. |
| `BI_ANALYTICS_BUILD_TS` | (auto) | Время сборки (ISO-8601). Используется только для бейджа версии в sidebar. |

### Версионирование и автоматическая инвалидация старого деплоя

Чтобы dev/release/localhost всегда показывали данные **именно текущей версии кода**, в системе добавлен идентификатор сборки `app_version` (`app_version.py`). Он меняется при каждом коммите (или при ручной смене `BI_ANALYTICS_BUILD_VERSION`). Поведение:

1. **`auto_ingest`** при старте читает маркер `.auto_ingest_done.txt` — если в нём `app_version` отличается от текущего, **принудительно** перевыкачивает данные (`force ingest`), удаляет `web_data.db` (если включён `BI_ANALYTICS_AUTO_INGEST_PURGE_DB`) и очищает `st.cache_data` / `st.cache_resource`. Так клиент после деплоя НЕ видит данные, построенные предыдущей версией кода.
2. **`project_visualization_app.py`** на каждом rerun сравнивает `_bi_app_version_sha` в `st.session_state` с текущим `app_version.sha`. При расхождении — `cache_data.clear()` + `cache_resource.clear()` + сброс session-кэшей (`_dev_matrix_cache_v1` и т.п.). Это страхует ситуацию «Streamlit перезагрузил воркер, но не убил процесс — кэш в памяти остался».
3. **Бейдж версии** в sidebar (внизу) показывает текущий sha и время сборки. Если у клиента видна старая версия — это будет видно в бейдже сразу.
4. **Кнопка «Обновить данные и кэш»** в sidebar (только admin/superadmin) форсирует `BI_ANALYTICS_AUTO_INGEST_FORCE=1` + `auto_ingest` + чистит кэши и перезагружает страницу.

Рекомендованный набор для **production** (release) и **dev**:

```toml
BI_ANALYTICS_AUTO_INGEST = "1"
BI_ANALYTICS_AUTO_INGEST_FTP = "1"
BI_ANALYTICS_AUTO_INGEST_AGE_H = "1"   # каждый ребут / каждый час, что наступит раньше
BI_ANALYTICS_AUTO_INGEST_PURGE_DB = "1"
```

### Чем отличается от release-only `BI_ANALYTICS_AUTO_FTP_ON_START`

| Аспект | Release-режим (`AUTO_FTP_ON_START`) | Общий флаг (`AUTO_INGEST`) |
|---|---|---|
| Когда срабатывает | при первом **заходе клиента** в release-режиме (по сессии) | при **холодном старте процесса** (любой режим) |
| Сессионная защита | да (`_release_web_autoload_tried`) | да (in-process) + дисковый маркер с TTL |
| Применимо в dev | нет (release-only) | да (если задать переменную) |
| Покрывает локально | нет | да (если запустить `streamlit run` с переменной в env) |

Оба механизма независимы и могут сосуществовать: первый страхует «клиент впервые открывает release-приложение», второй — «контейнер пересоздан, БД пустая».

### Логи

Все события auto-ingest пишутся в **stderr** инстанса (видно в Streamlit Cloud → «Manage app» → «Logs»):

```
[auto_ingest] START (no marker)
[auto_ingest] ftp_sync: downloaded=4, skipped_same_size=23, errors=0
[auto_ingest] load_all_from_web: loaded=27, skipped=0, version_id=318
[auto_ingest] DONE
```

Если что-то упало — приложение **всё равно запустится**, в логах будет:

```
[auto_ingest] load_all_from_web exception: ...
```

И UI откатится к ручной загрузке (как до фичи).

### Как включить на Streamlit Cloud

В Settings → Secrets добавить (вместе с уже существующей секцией `[ftp]`):

```toml
BI_ANALYTICS_AUTO_INGEST = "1"
# BI_ANALYTICS_AUTO_INGEST_AGE_H = "6"   # опционально: чаще обновлять
```

После сохранения — Streamlit Cloud сам перезапустит инстанс, ingest сработает на холодном старте.

### 4. Локально (разработчик)

Вариант A — задать те же `BI_FTP_*` через системные env / `.env` и в UI «FTP → web/ → Скачать…».

Вариант B — раз в день вручную скопировать файлы с FTP в локальный `web/` (winscp/scp/rsync). После — кнопка «Из папки web/ → Загрузить из web/».

В обоих случаях локальная `web/` будет совпадать с dev/release.

### 5. Сверка одинаковых цифр на трёх окружениях

После загрузки на каждом окружении сравните:
- в шапке: `Версия данных: YYYY-MM-DD HH:MM:SS | файлов: N, строк: M` — числа `N` и `M` должны совпадать;
- внизу любого дашборда (например, «Девелоперские проекты» → выбрать «Все»): набор проектов и значения вех должны быть одинаковыми.

Если расходится — значит FTP-синхронизация на каком-то окружении не отработала или env-переменные заданы по-разному. Проверьте `Settings → Secrets` на Cloud и системные env локально.

## Чек-лист деплоя на dev

1. На сервере:
   ```
   # Никаких release-флагов.
   ```
2. Видна dev-диагностика, доступен ручной режим загрузки **всем** ролям, демо-файлы (`sample_*`) **по умолчанию НЕ подмешиваются** — разработчик и тестировщик сразу видят данные так же, как клиент на release. Это безопасный дефолт.

3. **Admin может включить демо без перезапуска** — в сайдбаре под «Параметры отчётов» появляется блок «Dev-инструменты» с тумблером «Подмешивать демо-данные» (по умолчанию **выключен**). Включение → `sample_*.csv` и файлы из `new_csv/` начинают подмешиваться при загрузке из `web/`, и демо-строки остаются видимыми в матрице (например, «Дмитровский» / «Завод» из `sample_project_data_fixed.csv`). Полезно для UI-демонстрации без боевых данных. Действует на текущую сессию admin; на release блок скрыт.

4. Если нужно автоматическое включение демо для всех (например, на dev-стенде без боевых данных в `web/`), задайте:
   ```
   BI_ANALYTICS_INCLUDE_DEMO=1   # подмешивать демо по умолчанию для всех ролей
   ```
   Admin может через тумблер выключить демо себе на сессию (`_admin_demo_pref="ignore"` побеждает env).

## Smoke-тест поведения

```
venv\Scripts\python.exe scripts\_qa_release_mode_smoke.py
```

Скрипт проходит сценарии env + сценарии admin-сессионного тумблера и для каждого печатает:
- `is_release_client_mode`,
- `ignore_demo_data_files`,
- (для env-сценариев) сколько файлов вернёт `scan_web_files` и сколько среди них demo.

Корректное поведение:

| Сценарий | release | ignore_demo |
|---|---|---|
| baseline (нет env, дев) | False | **True** (новый безопасный дефолт) |
| `HIDE_DEV_DIAGNOSTICS=1` | True | True |
| `RELEASE_MODE=1` | True | True |
| **release + `INCLUDE_DEMO=1`** | **True** | **True** (override НЕ работает на release — это правильно) |
| dev + `IGNORE_DEMO=1` | False | True |
| dev + `INCLUDE_DEMO=1` | False | False |
| dev + admin pref `'include'` | False | False |
| dev + admin pref `'ignore'` | False | True |
| dev + admin pref `''` (не трогал) | False | True (тот же дефолт «как клиент») |
| release + admin pref `'include'` | True | True (admin не может включить демо на release) |
| dev + ENV `IGNORE_DEMO=1` + admin pref `'include'` | False | False (admin побеждает env) |
