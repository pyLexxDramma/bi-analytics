## 2. Источники данных (подтверждено по содержимому файлов)

|---|---|
| `web/AI/other_<DD-MM-YYYY>_resursi.csv` | Один файл = один месяц. Колонки-даты (например `01.04.2026..30.04.2026`) сгруппированы в `1 неделя..6 неделя`. Колонки шапки: `ID Проекта`, `Наименование проекта`, `Подрядчик` (или `ID Подрядчика + Подрядчик_new`), `Тип ресурсов ∈ {рабочие, техника}`. Заголовки на строке 1 (header_row=1). Snapshot берётся «не позже даты последнего файла ресурсов». Также берём `Наименование_Договора`, `Дата_Начала_Договора`, `Дата_Окончания_Договора`, `Сумма_Договора`. ## 3. Структура отчёта (новая)

Один заголовок «**ГДРС**» с **3 табами** (вместо нынешних 4):

### Таб 1. «Таблица» (Скрин 11) — главный вид

Шапка: `График движения рабочей силы (<люди|техника>), <Месяц> <Год>` + дата файла справа (формат `DD.MM.YYYY`).

Группировка: **Проект → Контрагенты внутри проекта → строка «Итого» по проекту → строка «Итого» суммарная**.

Колонки (12):

| # | Колонка | Источник |
|---|---|---|
| 1 | Контрагент | `Подрядчик` / `Наименование_Контрагента` |
| 2 | Вид работы | `Наименование_Договора` (или из CSV если есть) |
| 3 | План | `Dogovor.Количество_Людей` (или `_Техники`) → fallback `spravochniki.КоличествоРаботников/СпецТехники`. По умолчанию суммируем, если несколько договоров |
| 4 | СКУД | СКУД-факт = SUM ежедневных значений за весь месяц / N дней с данными (среднее за день в месяце) |
| 5 | Отклонение | `План − СКУД` (>0 = факт ниже плана = красный шрифт; ≤0 = зелёный) |
| 6 | 1 неделя | Среднее за день внутри 1-й недели (SUM/days_in_week) |
| 7 | 2 неделя | то же |
| 8 | 3 неделя | то же |
| 9 | 4 неделя | то же |
| 10 | 5 неделя | то же |
| 11 | 6 неделя | то же |
| 12 | Дельта (%) | `(План − СКУД)/План × 100` |

Доп.требования:
- Heatmap «Дельта (%)»: тёмно-красный ≤ −80%, светло-красный (−80..−30), розовый (−30..0), светло-зелёный ≥ 0.
- В строке «Итого по проекту» / «Итого общее» — формулы Σ.
- Логотип НЕ добавлять (по указанию пользователя 2026-05-07).

### Таб 2. «Динамика» (Скрин 3, 7) — графика

Сверху — line chart (тонкая линия, светло-голубой = план, рыжий = факт): «Фактическое количество по периодам (точки = недели): Люди (ресурсы)». Селектор оси X: **неделя / месяц / год**.

Под ним — для каждого выбранного проекта:
- **Pie «План/Факт по проекту»** (большой круг). Если выбрано несколько — pie-charts в строку. Сверху каждого — название проекта; справа — мини-сводка `План | Факт | Отклонение`.
- Цвет индикатора Отклонения: ≤0 → зелёный (хорошо), >0 → красный (плохо).
- На pie проценты-факта показывать **внутри** круга, а названия секторов — **снаружи**, без указания если значение есть в основном план/факт по оси X.

Под pie — **bar «План, Среднее за месяц, Дельта (группировка по контрагентам)»** (Скрин 4):
- 3 столбца на контрагента: План (голубой), Факт (зелёный если ≥ Плана), Дельта/Отклонение (оранжевый если >0, то есть план не выполнен; зелёный если ≤0).
- Сортировка по убыванию `План` слева направо. При выборе через легенду только одной серии — сортировка пересчитывается по выбранной серии.
- Все значения — выше оси X, цвет диаграммы определяет знак.

Под bar — **`% фактический по подрядчикам`** (Скрин 3 нижний блок) — столбцы «факт/план × 100%» с подписью значения в скобках.

### Таб 3. «Сводка по контрагентам» (Скрин 5) — таблица + метрики

- Таблица: `Контрагент | План | Среднее за месяц | Отклонение`.
  - «Отклонение» = `План - Среднее за месяц`. Шрифт: >0 → красный, ≤0 → зелёный.
- Под таблицей — 3 метрики: `Общий план | Объём средних значений | Общее отклонение`.
- «Общая дельта» переименовать в «Общее отклонение».

## 6. Известные неоднозначности (требуют ack пользователя)

1. **Источник «Плана»**: `Dogovor.Количество_Людей` обычно `None` → fallback на `spravochniki.КоличествоРаботников`? **Принимаемая гипотеза:** да, через fallback.
2. **Распознавание «месяца» отчёта**: из имени файла `other_<DD>-<MM>-<YYYY>_resursi.csv` (например `01-05-2026` → апрель 2026; `30-03-2026` → март 2026; в датах CSV сами столбцы это подтверждают).
3. **Если фильтр «Период от-до»** охватывает несколько месяцев — что в Pie показывать? **Принимаемая гипотеза:** объединённый план (Σ за месяцы) и объединённый факт (Σ за дни в диапазоне) / число дней.
4. **«Вид работы»** в новой таблице — на скрине ТЗ это длинная свободная строка («Вынос сетей», «Газоразрядка котельной», «СМР Блок», «АУПТ», «ЗОМ + ГРЩ»…). В наших данных её НЕТ — ни в `resursi.csv`, ни в `Dogovor.json`. **Гипотеза:** оставить эту колонку с подписью «—» (или скрыть), пока заказчик не пришлёт файл-источник видов работ.
5. **Старая логика `dashboard_workforce_movement` / `dashboard_technique_tabs` / `dashboard_gdrs_equipment`** — удалять полностью или сохранить как `dashboard_gdrs_legacy` для бэкап-просмотра?
