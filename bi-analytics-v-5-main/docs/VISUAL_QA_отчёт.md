## Журнал замечаний (по скринам)

- **п.1 (данные):** визуально ок; корректность метрик/таблиц — **открыто**, проверять по **§6** (`Отчеты для сверки.xlsx`) и согласованным эталонам.
- **п.2 / круг «Доли причин по проекту» (2026-04-25):** жалоба — мелкие радиальные подписи (название + число + %) внутри секторов. **В коде:** donut, на диаграмме только **%**; при **≤10** причин — процент **снаружи** (коннектор), иначе **внутри** горизонтально; полные названия — в **легенде**; реестр **53**. После обновления приложения — перепроверить скрином.
- **п.2 (дожим):** «тесно» — временно **топ‑9 + Прочее** (**реестр 54**); затем запрос: **все доли**, % с **длинным выносом** — снова полный набор сегментов, `domain` + крупные поля, только % снаружи (**реестр 55**).
- **п.2 (итог 2026-04-25):** заказчик — про усечение шкалы и сетку периодов; визуально **пока ок**.

## График проекта — что учитывать в визуализации (обсуждение с заказчиком)

- **Баланс масштаба:** увеличение высоты контейнера не должно автоматически умножать шрифты/толщины линий/маркеры в той же пропорции.
- **Плотность подписей:** для больших выборок нужен адаптивный режим (`summary-only`/`smart step`/`hover-only`), иначе неизбежны наложения.
- **Иерархия задач:** длинные названия и уровни WBS должны иметь ограничение длины и предсказуемое усечение, без «разваливания» левого столбца.
- **Режимы графика:** «Гантт», «Линии дат», «Ковенанты» должны визуально вести себя согласованно по высоте, маржинам и шрифтам.
- **Границы рендера:** высота графика не должна принудительно клампиться (`max_height`) при большом числе строк.
- **Поля и хвост по X:** правый отступ и диапазон дат должны считаться адаптивно, чтобы подписи конца полос не обрезались и не уезжали в «пустоту».

### Типовые некорректные визуальные баги при разных логиках

- **Логика «всё масштабировать xN»:** огромные подписи задач/дат, наложение текста на соседние строки, визуальный шум.
- **Логика «все inline-подписи всегда включены»:** пересечения дат/процентов на плотных графиках, нечитаемые каскады текста.
- **Логика «фиксированная малая высота»:** слипшиеся строки, маркеры перекрывают бары, потеря читаемости по оси Y.
- **Логика «фиксированный большой правый tail»:** ощущение пустого графика справа и смещение визуального фокуса.
- **Логика «жёсткий clamp по max_height»:** фактическая высота урезается, подписи/бары оказываются сжатыми, появляются артефакты плотности.
- **Логика «без учёта режима иерархии»:** дубли/скачки в позиционировании подписей, некорректное восприятие вложенности задач.
