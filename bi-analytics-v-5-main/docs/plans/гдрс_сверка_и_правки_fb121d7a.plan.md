## Чеклист из PDF (текст стр. 1–5; макеты — по рисункам в файле)

**Наименование и данные**

- Отчёт переименовать в **«ГДРС»** с пояснением в скобках *(график движения рабочей силы)*.
- Фактические данные — из файла вида **`ресурсы_*_CSV`** (пример в PDF); учитывать **ежемесячные версии** и **хранение истории** по версиям для сравнения через ИИ-команды (архив снимков / метаданные загрузки).

**Таблица «Рабочие» (форма Рисунок 1 в PDF)**

- **Контрагент** — из ресурсов CSV (колонка контрагента).
- **Начало**, **Окончание**, **План** — из **«Договор CSV»** по договору контрагента (в коде — сопоставить с уже загружаемыми договорными/справочными источниками).
- **Факт (среднее значение/день)** — из ресурсов CSV.
- **Отклонение** = **План − Факт** (явно в PDF; это **не** совпадает с текущим [16](D:/AI_codding/Analitics/bi-analytics-v-5-main/bi-analytics-v-5-main/docs/логика/16_ГДРС_рабочие.md), где указано Факт − План — после внедрения обновить markdown под заказчика).
- **Факт СКУД** — из ресурсов CSV: среднее по людям **по каждой неделе**.
- Колонку **«Дельта»** переименовать в смысл **(отклонение %)** = **Отклонение / План × 100%**.
- Колонка **«Отклонение %»** — условное форматирование от светло-зелёного до красного по величине % (как в Excel-эталоне страницы).
- Над контрагентами показывать **проект**; строки проекта — **суммарно** по правилам ячеек Excel с этой страницы.
- Нижняя строка таблицы — **«Итого»** (формулы как на странице Excel Рисунка 1).

**Фильтры (PDF)**

- Проект; Контрагент; **вид ресурсов**: рабочие | техника — из ресурсов CSV по признаку в **столбце H** (Рабочие / Техника), как на Рисунке 3.

**Графики и визуалы**

- Диаграмма **«Динамика по рабочим»** / **«Динамика по технике»** — в зависимости от фильтра; период оси абсцисс: **неделя / месяц / год**.
- Блок **план/факт по проекту** (круговая): увеличить круг; при нескольких проектах — **в один ряд**, над каждым кругом название проекта; справа сводка план/факт/отклонение с цветовыми индикаторами (в тексте PDF есть противоречивые строки про красный/зелёный — **сверить с Рисунком и Excel**, при двусмысленности запросить уточнение у заказчика).
- Блок **% фактический по подрядчикам**: столбец **«Доля(%)»** заменить на **«%»** = **факт/план×100%**; добавить столбцы **план**, **отклонение** = **план − факт**; индикаторы/цвет шрифта по знаку отклонения (как в PDF п. 58–62).
- Столбчатая диаграмма по подрядчикам: исправить логику **%**; серии План / Факт / отклонение (+/-); порядок подрядчиков по убыванию **Плана** слева направо; при фильтре легенды только одной серии — сортировка по этой метрике; отклонения ≠ 0 — над осью абсцисс; заменить подпись **«Дельта»** на **«Отклонение»**, формула **План − Факт**; **«Общая дельта»** → **«Общее отклонение»**; один из визуалов помечен **«Этот визуал удалить»** — убрать после идентификации на скрине PDF.
- Отдельный визуал (стр. 7–9 PDF): нужен **и для людей, и для техники** — линии **плана** и **факта** разными цветами (**голубой** план, **рыжий** факт); **плановые % до целых = 100%**, факт — расчётно.

**Замена таблиц (стр. 11–14 PDF)**

- Все текущие таблицы отчёта заменить на эталонные из PDF (несколько макетов подряд — сверить каждый с текущими `st.dataframe`/HTML в [`dashboard_workforce_movement`](D:/AI_codding/Analitics/bi-analytics-v-5-main/bi-analytics-v-5-main/dashboards/_renderers.py) и в [`dashboard_technique`](D:/AI_codding/Analitics/bi-analytics-v-5-main/bi-analytics-v-5-main/dashboards/_renderers.py)).

## Выявленные расхождения с ТЗ (16/17) и маппингом — что синхронизировать

**1. Чекбоксы в docs/16 vs PDF и текущая навигация**

- **PDF (стр. 1):** фильтр **вида ресурсов** — «рабочие / техника» по **столбцу H** в CSV (не чекбоксы).
- **docs/16:** чекбоксы «Только рабочие» / «Показывать долю» — трактовать как **устаревшие относительно PDF**, если заказчик не требует дублировать оба паттерна.
- **Код:** два пункта меню + `data_source_filter` + при `None` radio «Вид ресурсов» — **привести к PDF:** единый фильтр по признаку колонки H (или эквиваленту после нормализации имён колонок в `web_loader`).
- «Показывать долю подрядчика» — в PDF заложены **доли и %** в отдельных блоках (стр. 2–4); реализовать отображение согласно макетам, а не как абстрактный чекбокс без связи с графиком.

**2. Фильтры — подписи и состав**

- **PDF:** Проект; **Контрагент**; вид ресурсов (рабочие/техника) по столбцу H.
- Код: «Фильтр по контрагенту» уже близко; при необходимости синхронизировать термин «Подрядчик» только если заказчик так именует поле в других отчётах.
- Фильтр **типа техники** (узже чем «техника»): если потребуется после сверки с Рисунком 3 — добавить уровень ниже «Техника» по колонке типа ресурса.

**3. Формулы метрик (приоритет PDF)**

- **PDF:** **Отклонение = План − Факт**; отклонение **%** = **Отклонение / План × 100**; отдельно в блоках долей фигурирует **факт / план × 100%**.
- **Текущий код / таблица СКУД:** часто **СКУД − План** и процент от отклонения — потребуется **переворот знака** и выравнивание названий колонок («Дельта» → «Отклонение», добавить явные **Начало/Окончание** из договора если есть источник).
- Внутренний [16](D:/AI_codding/Analitics/bi-analytics-v-5-main/bi-analytics-v-5-main/docs/логика/16_ГДРС_рабочие.md) после реализации привести в соответствие с PDF (или пометить «устарело»).

**4. Логика плана из JSON vs пункт ТЗ про «неделю месяца по дате»**

- ТЗ (п.30 [16](D:/AI_codding/Analitics/bi-analytics-v-5-main/bi-analytics-v-5-main/docs/логика/16_ГДРС_рабочие.md)): по `КоличествоРаботников.Дата` определяется **номер недели в месяце**, затем план по неделям.
- Текущий код: `_pick_plan_for_period` выбирает одно «активное» значение для строки по концу месяца периода из ступенчатого timeline — **без явного преобразования дат JSON в недели месяца**.
- **План:** уточнить по PDF/заказчику ожидаемую гранулярность плана в отчёте (месяц vs недели 1–6). Если нужны недели строго по ТЗ — доработать merge: для каждой строки факта и каждой недели брать действующее `{Дата, Количество}` и заполнять плановые недели (согласовать с уже используемой сеткой «1–6 неделя» в таблице).
