### Заметки

- Пустой JSON при сбросе = встроенный список `CONTROL_POINT_MILESTONES` из кода.

#### Как администратору настроить отчёт «Контрольные точки» (маппинг к CSV MS Project)

1. Войти под ролью **администратор** или **суперадминистратор**.
2. В боковом меню открыть **Общие настройки** (страница `pages/admin.py`).
3. Перейти на вкладку **«Контрольные точки (MSP)»**.
4. В поле JSON задать **массив вех**. У каждой вехи:
   - **`title`** — заголовок группы столбцов в отчёте (то, что видят пользователи).
   - **`slug`** — внутренний ключ (латиница, без пробелов); по нему строятся поля в данных (`slug_plan`, `slug_fact`, …). Менять slug после накопления данных стоит осторожно.
   - **`match`** — правила отбора строк из загруженного MSP (как в коде: `level`, `names_any`, `name_contains`, `parent_l2_contains`, `block_contains`, `phase_needles`, `phase_exclude_needles` и т.д.). Это и есть **сопоставление ячеек отчёта с задачами в CSV MS Project**.
5. Нажать **«Сохранить»**. Настройка пишется в БД по ключу **`control_points_milestones_json`**.
6. **«Сбросить на встроенные правила»** — очистить ключ и снова использовать вехи из кода (`CONTROL_POINT_MILESTONES`).
7. **«Скачать шаблон по умолчанию»** — эталонный JSON для правок.

#### Ошибки «Could not find page: …/pages/admin.py»

1. **Корневой** `streamlit_app.py` (Streamlit Cloud / локально) регистрирует multipage только из **`<корень репозитория>/pages/`**. Реальные страницы лежат во **вложенной** папке `bi-analytics-v-5-main/pages/`, поэтому в **корне** добавлены прокси `pages/admin.py`, `pages/profile.py`, `pages/analyst_params.py` — они выполняют `runpy` на соответствующие файлы внутри пакета приложения.
2. **`config.switch_page_app`** вызывает `st.switch_page("pages/...")` без префиксов и без абсолютных путей: путь всегда относительно каталога главного скрипта; при прокси в корне это совпадает с зарегистрированной страницей.

## Шаг 3 — «График проекта» (макет правок)

**Статус:** сделано (ожидает ручной проверки)

### Правка

**Файлы:** `dashboards/_renderers.py` (`dashboard_project_schedule_chart`, вспомогательные `_render_gantt_schedule_html_table`, …), `utils.py` (`apply_chart_background`), `project_visualization_app.py` (CSS сайдбара)

1. **Фильтры:** в одной строке — проект (ур. 1), функциональный блок (ур. 2), уровень отображения MSP (3 / 4 / 5 или **«Все уровни»** по умолчанию), режим подписей у концов полос (**дата окончания** / **% выполнения**). Ниже строкой — **«Показать причины отклонений»** и **«Отображать в лотах»** (если в данных есть колонка лота).
2. **Гант:** `px.timeline` для плана + второй trace для базы (`barmode=group`); подписи у концов полос — аннотации; ось X с русскими подписями дат/месяцев; при необходимости — опция «в лотах» и поиск колонки уровня/лота по алиасам.
3. **Ось Y:** отступы `margin.l` / `ticklabelstandoff`; **не** используется CSS `translateX` по ширине margin — иначе подписи задач уезжают за край и нечитаемы.
4. **Таблица** (expander «Таблица (первые строки)» под подписью к графику): фиксированный порядок колонок — **Проект**, **Задача**, **План начало** / **План окончание**, **База: начало** / **База: окончание**, при наличии — **% выполнения**, всегда — **Отклонение Начала** и **Отклонение Окончания** (поля отклонения MSP или разница план − база в днях, формат «±N дн.»). При включённом **«Показать причины отклонений»** — **Причины отклонений** и **Заметки** (если колонки есть в выгрузке). Подсветка ячеек отклонений (задержка / опережение) через `Styler`, как в блоке отклонений по дням.
5. **Plotly:** `apply_chart_background(..., skip_uniformtext=True)` для этого графика — иначе глобальный `uniformtext mode=hide` может скрывать подписи у полос.
6. **Навигация:** скрыт стандартный список страниц Streamlit (`[data-testid="stSidebarNav"]`); переходы — через меню приложения.

### Проверка (ручная)

- График проекта: фильтры, план/база двумя цветами, подписи у концов полос, таблица с нужными заголовками и отклонениями; при включении чекбокса — колонки причин/заметок.
- Подписи задач слева на диаграмме читаемы (не «полоска»).

## Шаг 4 — «Значения отклонений от базового плана»: фильтры по уровням MSP

**Статус:** сделано (ожидает ручной проверки)

### Правка

**Файл:** `dashboards/_renderers.py` — `dashboard_deviation_by_tasks_current_month`, вспомогательные `_dev_tasks_find_column`, `_dev_tasks_resolve_level_column`, `_dev_tasks_build_ancestor_keys`

1. Убран **«Фильтр по этапу»** (колонка `section`).
2. **Функциональный блок** — список задач **уровня 2** по колонке уровня MSP (имена задач с `Outline Level` / `level` = 2).
3. **Строение** — задачи **уровня 3**; при выбранном блоке список ограничивается строками, у которых предок ур. 2 совпадает с выбранным блоком (обход иерархии по порядку строк MSP).
4. **Детализация** — переключатель **уровень 4 (укрупнённо)** / **уровень 5 (детально)**; в выборку попадают только строки с соответствующим уровнем структуры.
5. Если колонка уровня не найдена — подсказка и fallback: при наличии колонок **Функциональный блок** / **Строение** в файле — фильтры по ним; без уровней 4/5.
6. Блок **«Детализация отклонений по задачам»** использует **те же** фильтры, что основной график; группировка по `task name` без «Раздела».

### Проверка (ручная)

- Экран «Значения отклонений от базового плана»: нет «Фильтр по этапу»; четыре колонки фильтров (проект, блок, строение, детализация) ведут себя по ТЗ на типичной выгрузке MSP.

### Заметки

- Иерархия считается по **порядку строк** в загруженном датафрейме (как в типичном экспорте MSP). При нестандартной сортировке исходного файла списки блок/строение могут потребовать уточнения.
