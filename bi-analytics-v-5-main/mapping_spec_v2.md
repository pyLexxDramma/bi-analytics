# mapping_spec_v2

## 1) Цель

Зафиксировать рабочую схему маппинга данных для BI-дашбордов с учетом:
- фактических полей выгрузок в `web/` и `web/AI/`;
- файла сверки `Правки/Отчеты для сверки.xlsx`;
- замечаний из двух видео встречи от `24.04.2026`.

Документ ориентирован на внедрение в текущий код проекта.

### 1.1 Где лежат данные в репозитории
- **`web/`** — рабочая папка выгрузок для приложения: 1С JSON, TESSA CSV, справочники; подкаталог **`web/AI/`** — MSP и «прочие» (`other_*_resursi`, `other_*_rd` и т.д.). Соглашения имён и типы файлов для ETL: **`etl/parser.py`** (докстринг в начале файла).
- **`etl/`** — `parser` / `mapper` / `loader` / `schema` (SQLite `etl.db` при использовании пайплайна). Чеклист п. **5.1**.
- **`new_csv/`** — примеры структуры для тестов/витрин; **не** являются эталоном для сверки с `Правки/Отчеты для сверки.xlsx` (эталон — фактические файлы в **`web/`** / **`web/AI/`**).
- **`Правки/Отчеты для сверки.xlsx`** — вне репозитория или копия рядом с `Правки/`; листы для приёмки — п. **6**.

## 2) Канонические источники и поля

- `Отклонение_начала`, `Отклонение_окончания`

Источник: `web/tessa_*_id.csv`

Ключевые поля:

- `1C_ID_PODR`, `1C_ID_CONTR`, `1C_ID_LOT`

### 2.3 TESSA RD (РД-факт)

### 2.4 TESSA Tasks (статусы задач, причина/результат)

### 2.5 1C финансы
Источники:
- `web/*_dannye.json` (БДДС/оборотные)
- `web/*_DK.json` (ДЗ/КЗ)
- `web/*_spravochniki.json` (справочники 1С; **для ГДРС** этот же файл — источник **плановых** значений по согласованию с заказчиком, см. **§2.6**)

Ключевые поля (минимум в текущих выгрузках; план ГДРС — расширение/иные узлы в том же `spravochniki` — по схеме 1С):
- `dannye.json`: `Период`, `Сценарий`, `СтатьяОборотов`, `Проект`, `Сумма`, `РасходДоход`, `ТипСтатьи`
- `DK.json`: `Организация`, `Контрагент`, `Договор`, `ОстатокНаНачалоПериода`, `ОстатокНаНачалоПериодаПоАвансам`, `ВсегоОплат`, `ВсегоОплат_Аванс`, `ОстатокНаКонецПериода`, `ОстатокНаКонецПериодаПоАвансам`
- `spravochniki.json`: `ID_Контрагента`, `Наименование_Контрагента`, `ИНН_Контрагента`, `КПП_Контрагента` (и **плановые показатели ГДРС** — в структуре выгрузки 1С; детализация полей «люди / техника» — фиксировать по фактическому JSON после полной поставки)

### 2.6 ГДРС/СКУД
**План (плановые значения, люди/техника):** `web/1c_*_spravochniki.json` (пример имени: `1c_27-03-2026_07-30_spravochniki.json`) — **источник плана** для ГДРС. Склейка с фактом: по `ID_Контрагента` / `Наименование_Контрагента` ↔ `Подрядчик` в `resursi` (с нормализацией имён, как в **§3.1**).

**Факт (СКУД, динамика по дням/неделям):** `web/AI/other_*_resursi.csv` (примеры: `other_01-01-2026_resursi.csv`, `other_01-02-2026_resursi.csv`).

Ключевые поля (факт):
- `Проект`, `Подрядчик` (в коде встречается и сопоставление с `Контрагент`);
- `тип ресурсов` — `рабочие` / `техника` (и варианты написания); на вкладке «ГДРС» (рабочие) в референсной таблице отчёта оставляются строки **только с рабочими**, техника выносится в меню **«ГДРС Техника»**;
- при наличии в выгрузке — колонка **`data_source`** (семантика: строки «Ресурсы» / «Техника») для согласования с вкладками; если колонки нет, тип определяется по структуре файла и `тип ресурсов` (см. `dashboard_workforce_movement`);
- ежедневные столбцы дат (`ДД.ММ.ГГГГ`), при необходимости — колонки **«N неделя»**, агрегаты вида `среднее значение ...`, `среднее за месяц` / `среднее за неделю` (имена — по фактическому файлу).

**Сверка:** листы Excel **`ГДРС (вычищен, факт, январь)`** / **`ГДРС (сводная, факт, январь)`** в `Правки/Отчеты для сверки.xlsx` — эталон агрегатов **факта**; план в приёмке — согласовать с полями в **`1c_*_spravochniki.json`** после внедрения чтения плановых величин в коде.

## 3) Сквозные ключи и правила связи

### 3.1 Приоритет ключа проекта
1. `MSP.ID_проекта = TESSA_RD.ObjectProjectID`
2. fallback по имени проекта (`ObjectName`/`Проект`) после нормализации

- удаление служебных кавычек/невидимых символов;

## 4.8 UI: «плотные колонки/столбцы» (таблицы + графики) — **на всех дашбордах**
- **Таблицы (HTML/`st.dataframe` где применимо):** не оставлять визуально «пустую» ширину колонок; для фиксированного layout задавать классы/ширины по смыслу (даты/ID/суммы — уже, текст — шире с переносом); избегать `min-width`, который растягивает таблицу «в пустоту`, если контента мало; горизонтальный скролл — нормально, но **цель** — у пользователя сразу видна «привязка» значения к заголовку.
- **Графики/диаграммы (в первую очередь Plotly):** оси/категории не должны выглядеть как «широкий зазор без причины»; для bar/horizontal bar — контролировать `bargap`, `margin`, длину подписей (перенос/угол), при необходимости `tickangle`, `tickformat`, `automargin` и/или `uirevision` (как сейчас в части отчётов) так, чтобы подписи и значения **читались без угадывания**, а легенда не «съедала» полезную площадь.
- **Единообразие:** при добавлении нового отчёта — повторять тот же UX‑принцип: компактные числовые/датовые поля, аккуратные длинные строки, предсказуемые отступы.

## 4.9 UI: без «серых» дебаг/служебных подсказок — **на всех дашбордах**
- **Не показывать** мелкий серый текст `st.caption` под заголовками/таблицами/графиками, если это служебное пояснение, «запись N», отладка источника данных или дублирует то, что уже на экране. В коде для production используется заглушка (`dashboards/ui_quiet.py`: `suppress_caption`) вместо `st.caption`.
- **`st.expander`** — не для дебаг-диагностики и длинных инструкций «как устроено внутри»; при необходимости пояснить — через согласованный с заказчиком блок (без дублей под графиками).
- **`st.info` / `st.warning` / `st.error`** — оставлять для реальных состояний (нет данных, ошибка загрузки, нарушение фильтра); это **не** относится к пункту про «серые» подсказки.
- **Про `help=` у виджетов** — отдельный пункт плана: **4.10** (снятие/согласование).

## 4.10 UI: снятие лишних `help=` с фильтров и виджетов — **отдельная задача внедрения**
- **Цель:** в интерфейсе нет «второй линии» пояснений через иконку `?` у `st.selectbox` / `st.multiselect` / `st.checkbox` и т.п., если это внутренняя памятка разработчику, дублирует лейбл или противоречит 4.9.
- **Допустимо:** краткие продуктовые `help=` строго по согласованному макету/ТЗ (редко, единообразно).
- **В коде:** пройти `dashboards/_renderers.py` (и при необходимости другие экраны), удалить/сжать многострочные `help=(...)`, внутренние пояснения про структуру MSP/иерархию — вынести в документацию поставщика данных, не в UI.

## 4.11 UI: подписи на **всех** диаграммах и графиках (видны сразу, порядок, читабельность)
- **Видимость:** основные подписи категорий, рядов, значений и ключевых меток должны читаться **с первого взгляда**, без обязательного наведения. **Hover** в Plotly — только дополнение (детализация, длинный текст), а не единственный способ понять, что на графике.
- **Расположение:** единая логика — не «лес» из налезающих лейблов; при необходимости: `text`/`textposition` на бар/точки, `tickangle` / перенос длинных подписей оси, `automargin`, предсказуемые `margin`, при пересечениях — прореживание тиков (`nticks`, `tickmode`, сокращение длинных подписей с `…` + полный текст в hover).
- **Числа и единицы:** формат осей и подписей над столбцами/секторами — в одном стиле с отчётом (разделители, `%`, сокращения); избегать наезда текста на ряд/столбец (`uniformtext`, разумный `minsize` или сокращение).
- **Легенда:** если ряды не очевидны с оси/подписей — легенда читаема, не перекрывает график (положение/строка/столбец по шаблону отчёта), цвета согласованы с макетом.
- **Новые отчёты:** копировать тот же подход, что в уже вылизанных графиках (см. вспомогательные `apply_chart_background`, `_apply_bar_uniformtext`, раскладки finance-баров и т.д.).
- **Код (централизация):** `render_chart` вызывает `_apply_plotly_spec_411_labels` (Bar: подписи значений, если не задан `text`/`texttemplate`, без Gantt/дат по эвристике; Pie: при отсутствии подписи — `textinfo` с `label+percent` при необходимости); в `utils.apply_chart_background` для `uniformtext` используется `mode="show"` (и согласовано с `_apply_bar_uniformtext` / рядом `update_layout` в отчётах).
- **Приёмка:** пункты §5.2 / §6 по 4.11 отмечены `[x]` после ответа **«ok»** (2026-04-24).
