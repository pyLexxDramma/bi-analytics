# mapping_spec_v2

## 1) Цель

Зафиксировать рабочую схему маппинга данных для BI-дашбордов с учетом:
- фактических полей выгрузок в `web/` и `web/AI/`;
- файла сверки `Правки/Отчеты для сверки.xlsx`;
- замечаний из двух видео встречи от `24.04.2026`.

Документ ориентирован на внедрение в текущий код проекта.

## 2) Канонические источники и поля

### 2.1 MSP (сроки, причины, ПД, контрольные точки)
Источник: `web/AI/msp_*_<date>.csv`

Ключевые поля:
- `ID_проекта`, `ЛОТ`, `БЛОК`, `Уровень`, `Название`, `Ид`, `Уникальный_идентификатор`
- `Базовое_начало`, `Базовое_окончание`, `Начало`, `Окончание`
- `Процент_завершения`, `Причины_отклонений`
- `Отклонение_начала`, `Отклонение_окончания`

### 2.2 TESSA ID (ИД и предписания)
Источник: `web/tessa_*_id.csv`

Ключевые поля:
- `DocID`, `KindName`, `KrState`, `KrStateID`, `CreationDate`
- `1C_ID_PODR`, `1C_ID_CONTR`, `1C_ID_LOT`
- `CONTR`, `ObjectName`, `Lot`, `DocNumber`, `DocDescription`

### 2.3 TESSA RD (РД-факт)
Источник: `web/tessa_*_rd.csv`

Ключевые поля:
- `DocID`, `KrState`, `CreationDate`
- `ObjectName`, `ObjectProjectID`, `ObjectProjectName`
- `DivisionCipher`, `SubDivisionVersionName`
- `CONTR`, `Lot`

### 2.4 TESSA Tasks (статусы задач, причина/результат)
Источник: `web/tessa_*_task.csv`

Ключевые поля:
- `CardID`, `TypeCaption`, `OptionCaption`, `Result`, `Completed`

### 2.5 1C финансы
Источники:
- `web/*_dannye.json` (БДДС/оборотные)
- `web/*_DK.json` (ДЗ/КЗ)
- `web/*_spravochniki.json` (справочник контрагентов)

Ключевые поля:
- `dannye.json`: `Период`, `Сценарий`, `СтатьяОборотов`, `Проект`, `Сумма`, `РасходДоход`, `ТипСтатьи`

- `spravochniki.json`: `ID_Контрагента`, `Наименование_Контрагента`, `ИНН_Контрагента`, `КПП_Контрагента`

### 2.6 ГДРС/СКУД
Источник: `web/AI/other_*_resursi.csv`

Ключевые поля:
- `Проект`, `Подрядчик`, `тип ресурсов`
- ежедневные столбцы дат
- агрегаты `среднее значение ...`

### 2.7 РД-план
Источник: `web/AI/other_*_rd.csv`

Ключевые поля:
- `Номер`, `Шифр`, `Наименование раздела`
- `Количество разделов по Договору`
- `Дата выдачи разделов по Договору`

## 3) Сквозные ключи и правила связи

### 3.1 Приоритет ключа проекта
1. `MSP.ID_проекта = TESSA_RD.ObjectProjectID`
2. fallback по имени проекта (`ObjectName`/`Проект`) после нормализации

Нормализация строкового ключа:
- trim, collapse пробелов, casefold;
- удаление служебных кавычек/невидимых символов;
- дедупликация дублей названий проекта в фильтрах.

### 3.2 Предписания/ИД: связь карточки и задачи
- Базовая связь: `TESSA_TASK.CardID = TESSA_ID.DocID`
- При неполных строках применять объединение по нормализованному ключу DocID/CardID.

### 3.3 РД-план ↔ РД-факт
- Ключ связи: `Проект + DivisionCipher(Шифр) + SubDivisionVersionName(Наименование раздела)`
- Если нет `Дата выдачи разделов по Договору` в плане, строку не показывать в плановом слое графика РД.

### 3.4 Статусы
- Расшифровка статусов: через `KrStates.csv`
- Для предписаний исключать служебный статус `KrState = Проект` из KPI и таблиц.

## 4) Маппинг по отчетам (что исправить по видео)

## 4.1 Контрольные точки / График проекта / Сроки
- Исправить маппинг проекта (убрать дубли `Дмитровский*` в фильтрах и агрегатах).
- Для дат использовать только пары:
  - План: `Базовое_начало`, `Базовое_окончание`
  - Факт: `Начало`, `Окончание`
- В таблицах переименовать плановые поля в базовые.
- Добавить/починить сортировку отклонений по клику заголовка.
- Починить фильтр уровней (2/3/4/5) и чекбокс скрытия завершенных задач.

## 4.2 Причины отклонений
- Единые фильтры: `Проект`, `ФБ/БЛОК`, `Строение/уровень`, `Период`, `ТОП-N причин`.
- Корректная обработка периода с частичными данными (без падений).
- Привести таблицу и stacked-диаграммы к логике сверки (подписи/доли/сортировка).

## 4.3 ГДРС и СКУД
- Убрать дублирующий экран/таб (`СКУД по неделям`).
- Базовый экран движения рабочей силы строить по `тип ресурсов = рабочие`.
- Иерархия таблицы: `Проект -> Подрядчик`.
- Отклонение считать как `Факт(СКУД) - План`.

## 4.4 Проектные работы (РД/ПД)
- Добавить статус `Передано подрядчику`.
- Поле "Задача" для РД брать из разделов TESSA/RD (а не из произвольного текстового поля).
- Добавить мультивыбор проектов.
- Для круговой/линейной РД:
  - добавить доли `Не принято` и `Не выдано`;
  - старт линии с 0;
  - цвета: факт оранжевый, план синий;
  - подписи % и количество просрочки.

## 4.5 Исполнительная документация
- Перепроверить заполнение KPI и таблиц через цепочку `tessa_id + tessa_task + KrStates`.
- Цвета статусов табов привести к ТЗ.
- Даты в UI показывать без времени.
- В фильтрах оставить обязательный чекбокс: "не показывать просрочку, если ИД сдана".

## 4.6 Неустраненные предписания
- Добавить дату выдачи и диапазон дат в фильтры.
- KPI пересчитать по единой логике:
  - всего;
  - устраненные;
  - просроченные;
  - неустраненные;
  - критические;
  - дни просрочки.
- Исключать `KrState=Проект`.
- Дедуп по ключу предписания (`DocID`, fallback `CardID`) обязателен.
- Стабилизировать повторный расчет после refresh без новой выгрузки.

## 4.7 Финансы
- В `DK` использовать расчетные поля:
  - `ОстатокНаНачало = ОстатокНаНачалоПериода - ОстатокНаНачалоПериодаПоАвансам`
  - `ОстатокНаКонец = ОстатокНаКонецПериода - ОстатокНаКонецПериодаПоАвансам`
- В "Прогнозном бюджете" убрать нижний лишний график (по замечанию заказчика).

## 4.8 UI: «плотные колонки/столбцы» (таблицы + графики) — **на всех дашбордах**
- **Таблицы (HTML/`st.dataframe` где применимо):** не оставлять визуально «пустую» ширину колонок; для фиксированного layout задавать классы/ширины по смыслу (даты/ID/суммы — уже, текст — шире с переносом); избегать `min-width`, который растягивает таблицу «в пустоту`, если контента мало; горизонтальный скролл — нормально, но **цель** — у пользователя сразу видна «привязка» значения к заголовку.
- **Графики/диаграммы (в первую очередь Plotly):** оси/категории не должны выглядеть как «широкий зазор без причины»; для bar/horizontal bar — контролировать `bargap`, `margin`, длину подписей (перенос/угол), при необходимости `tickangle`, `tickformat`, `automargin` и/или `uirevision` (как сейчас в части отчётов) так, чтобы подписи и значения **читались без угадывания**, а легенда не «съедала» полезную площадь.
- **Единообразие:** при добавлении нового отчёта — повторять тот же UX‑принцип: компактные числовые/датовые поля, аккуратные длинные строки, предсказуемые отступы.
